201 .thp files
202 ==========
203 
204 Version 1.0 (20050515)
205 
206 .thp is a video format on the gamecube. The video frames are independent "quasi-jpegs", and if audio frames are present, they are in an adpcm format (described below).
207 
208 
209 Header data
210 -----------
211 
212 The file starts with a thp header:
213 
214 
215 struct ThpHeader
216 {
217  char tag[4]; //'THP\0'
218 
219  u32 version; //0x00011000 = 1.1, 0x00010000 = 1.0

221  u32 maxAudioSamples; //!= 0 if sound is stored in file, maximal number of samples in one frame.
222  //you can use this field to check if file contains audio.
223 
224 
225  float fps; //usually 29.something (=0x41efc28f) for ntsc
226  u32 numFrames; //number of frames in the thp file
227  u32 firstFrameSize; //size of first frame (header + video + audio)
228 
229  u32 dataSize; //size of all frames (not counting the thp header structures)
230 
231  u32 componentDataOffset; //ThpComponents stored here (see below)
232  u32 offsetsDataOffset; //?? if != 0, offset to table with offsets of all frames?
233

274 Frame data
275 ----------
276 
277 A frame is made up of a frame header followed by a video frame followed by ThpAudioInfo.numData audio frames (only if the video contains sound).
278 
279 The frame header consists of 3 (or 4, if the video contains sound) u32's:
280 struct FrameHeader
281 {
282  u32 nextTotalSize; //total size of NEXT frame (frame header, video and audio)
283  u32 prevTotalSize; //total size of PREVIOUS frame
284  u32 imageSize; //size of image frame of THIS frame
285  (u32 audioSize; //size of one audio frame of THIS frame) <- only if the file contains audio
286 };
287 
288 
289 Directly after the frame header FrameHeader.imageSize bytes video information follow. Directly after the video information, ThpAudioInfo.numData audio frames follow, each Frameheader.audioSize bytes large (only if the file contains audio).
290 
291 Video Frames
292 ------------
293 
294 A video frame is more or less a jpeg image. A jpeg file is structured by several markers. A marker is a two-byte code, the first of the two bytes is 0xff. The jpeg standard states that if you want to store the value 0xff, you have to store it as 0xff 0x00 (else it would be confused with a marker). This is NOT the case in .thp files, the value 0xff is stored simply as 0xff in the image data. So if you want to use jpeglib to read the frame, you have to convert the thp "quasi-jpeg" to a real jpeg by converting 0xff values to 0xff 0x00 in the image data. You have to be careful that you don't convert the terminating End-Of-Image marker, though.
295 
296 - search for Start-Of-Image marker (0xff 0xda)
297 - search for End-Of-Image marker (0xff 0xd9) (start search at end of buffer and search backwards!)
298 - convert each 0xff between image data start and image data end to 0xff 0x00
299 - the resulting buffer can be passed to jpeglib to let it decode the image for you
